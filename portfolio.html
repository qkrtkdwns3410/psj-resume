🔐 Race Condition 해결

문제 상황

광고 시청 후 포인트 적립에서 동일 사용자의 동시 요청으로 인한 포인트 중복 적립 발생

sequenceDiagram participant U as User participant A as Application participant DB as Database Note over U,DB: Race Condition 발생 시나리오 U->>+A: Request 1: 포인트 적립 U->>+A: Request 2: 포인트 적립 (동시) A->>+DB: SELECT 현재 포인트 (Request 1) A->>+DB: SELECT 현재 포인트 (Request 2) DB-->>-A: 포인트: 1000 (Request 1) DB-->>-A: 포인트: 1000 (Request 2) A->>+DB: UPDATE 포인트 = 1100 (Request 1) A->>+DB: UPDATE 포인트 = 1100 (Request 2) DB-->>-A: 성공 (Request 1) DB-->>-A: 성공 (Request 2) Note over A,DB: 결과: 200포인트가 적립되어야 하지만
100포인트만 적립됨

해결 방안: 비관적 락(Pessimistic Lock)

  • 선택 이유: 금융 시스템의 강력한 데이터 정합성 요구사항
  • 구현 방식: SELECT ... FOR UPDATE를 통한 레코드 락 획득
  • 프로세스: 트랜잭션 시작 → 광고 시청 기록 → 비관적 락 획득 → 포인트 적립 → 트랜잭션 종료

읽기/쓰기 트랜잭션 분리
graph TD subgraph "Write Operation" A[Controller] --> B["Service (readOnly=false)"] B --> C[Master DB] end subgraph "Read Operation" D[Controller] --> E["Service (readOnly=true)"] E --> F[Slave DB] end style B fill:#ffe6e6 style E fill:#e6f3ff
REQUIRES_NEW 전파 속성을 이용한 독립 트랜잭션
sequenceDiagram participant C as Caller Service participant S as Callee Service participant DB as Database C->>S: Method A (@Transactional) activate S S->>DB: Transaction A Start S->>S: Method B (@Transactional(propagation=REQUIRES_NEW)) DB->>S: Suspend Transaction A S->>DB: Transaction B Start DB->>S: Transaction B End S->>DB: Resume Transaction A DB->>S: Transaction A End deactivate S
구현 상세
  • 읽기 전용 서비스:
    • @Transactional(readOnly=true, propagation=REQUIRES_NEW)
    • 별도의 트랜잭션으로 분리하여 Slave DB 연결 대비
    • REQUIRES_NEW로 기존 쓰기 트랜잭션과 독립적 실행
  • 쓰기 서비스:
    • @Transactional(readOnly=false)
    • Master DB 연결 대비 일반 트랜잭션
    • 롤백 시 데이터 일관성 보장
  • 전파 속성 분리: 읽기/쓰기 작업의 명확한 트랜잭션 경계 설정

기본 해결 방안

  • @EnableTransactionManagement: 글로벌 트랜잭션 관리 활성화
  • @Transactional 명시적 적용: 주요 서비스 메서드에 트랜잭션 경계 설정
  • 트랜잭션 매니저 검증: 데이터베이스 작업 시 적절한 트랜잭션 경계 확인
  • 롤백 테스트: 의도적 오류 발생으로 롤백 동작 검증
